Reflection:

Problems I encountered:

The main problem I encountered was that I previously had the price of the pizza's as strings. This lead to many errors and crashes as I couldn't properly calculate the cost as strings are not able to be multipled/added/subtracted. To fix this I changed the prices to floats (numbers) so they could be treated as numbers, which allowed my code to work. In theory, I changed the "18.5" to 18.5. Overall, it was a bad idea from the beginning to have the price as a string as there was no way any calculation would've worked. Since I tested this function indivually in sprint 5, I was able to change the prices to floats so total cost could be calculated properly.

I had 'order_cost = 18.5 * quantity_pizza'. This is a problem because gormet pizza's were different prices at $21.50. Also, if the pizza prices were going to change in the future, it would not be easy to update. For the first couple of tests I didn't notice this was a problem as I only ordered the standard pizza's. I fully realised how big of an issue it was when I ordered a gormet pizza ($3 more than a standard pizza) as it made the order cost completely wrong, which ended up making the totalcost wrong too. To fix this I had to find the cost of the exact pizza the customer ordered. I did this by using 'cost = plist[type_pizza][0]' which accessed the pizza list to find the correlating price to the type of pizza ordered. Once I had a value for cost (for the individual pizza either 18.5 or 21.5) I could accurately calculate the cost for a single pizza order using 'order_cost = cost * quantity_pizza'.

When a customer who hasn't ordered anything, tries to review their non-existent order, after the encouraging message to order the programme prints out part of the summary of pizza's ordered anyway. This is a problem as it would be confusing to the customer and theis information is redunant as if the customer hasn't ordered, then they wouldn't have a final cost or quantity. User Story

I asked tommi to order some pizza to see how an actual order would take place. At the beginning it was working smoothly but I realised that my program severely lacked the feature of changing one the pizza order. In this instance, Tommi wanted to change the quantity of the of her Pepperoni however I had no way to accomodate her. This made me realise that an update feature is very important to this program. Although I had planned to work on the get customer details function, I realised that an update function would be more beneficial to customer. This is the reason why I decided to prioritise this feature.

Reflection:

I made a couple of mistakes while completing this sprint; having strings for the prices instead of floats and using the standard cost of 18.5 for everything. However, due to the fact I was trialling and testing this component before incorporating it into the main program, I was able to catch all of these mistakes before they made my main crash. Testing this sprint made sure that all my costs in the final outcome were accurate, essential for a pizza ordering program. When testing my program when implemented in the main function, I found no problems. I then commited this sprint to Github.

I'm happy with the sucess of this sprint as I was able to make sure that the total_quantity and total_cost were calculated every time the review function is called, this means that if the user orders pizza's a couple of seperate times, the total_cost and total_quantity will still be accurate.

However, I haven't calculated GST yet, but I intend to do this in a later sprint. (this ended up being sprint 9). What am I going to focus on next? As i said before I will add the update feature, which I recently founf out was very important from Tommi's user test.
