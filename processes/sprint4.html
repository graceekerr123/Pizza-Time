Aim:

In this sprint I want to fix some more of the major problems that occured in sprint 2. Therefore the aim for sprint 4, is create functions to validate string entrys so the program won't crash.
I need to make a function which properly validates the entry of strings. For instance, when the user is asked to pick a letter from the menu as well. Secondary, when the user is asked to enter Y or N when ordering another pizza. Both of these are single character inputs, so that's what my validation function will have to address.


What are the problems I found in sprint 4:

1. A integer can be accepted as a string



2. Lowercase letters were not accepted, only uppercase were
3. If a space was added in the input, the program would always see the input as invalid



4. If two letters were inputed then it would be invalid

Reflection:

Iteriative decision for string validation:

Option 1:

This is the more basic string validation function. Some positive points is that it can validate a long input as well as small inputs. When using it in my code I use 'option = get_string("Please enter an option: ->", 1,1)' so it can only be one string, this is a problem as it doesn't accept an input like "RR" when the customer clearly wants the review menu. Another letter not part of the valid answers like "Z" or a number will also be accepted as there is nothing in the function to stop it.


Option 2:

This is a more advanced validation function. Negatively, it can only validate inputs of one character. However, it is very flexible to the situation as I include a char_list as one of my arguments. Therefore when I use this function it will look like this -'option = get_one_string2("Please enter an option: ->", ["P", "O", "R", "Q"])'. This is a huge advantage because only P, O, R, Q will be accepted, making sure that if another letter like Z or a number to be entered, it will ask the user to reenter.


Evaluation:

I chose to use option 2 because it is better suiting to the situations I have, which are both single inputs. At the moment I'm only trying to fix the problems at hand, if other situation occur where I need a string validation for a long string I will reassess. Option 2 fixes more errors that Option 1, e.g. RR will be accepted as R in Option 2 but it will not in Option1.
